# License Scanning with Trivy

Kreate's license scanning integration allows you to automatically verify the licenses of your project's dependencies against a list of forbidden or restricted licenses. This is critical in commercial environments to ensure legal compliance (e.g., avoiding "copyleft" licenses like GPL in proprietary software).

## How it Works

The `license { }` block defines how Trivy analyzes your project for license compliance. Kreate supports two distinct scanning modes:

### 1. Lockfile Scan (Default)
This is the default mode. Trivy specifically analyzes the `.lockfile` files generated by Gradle.
*   **Pros**: Extremely fast and highly precise for identifying package-level licenses.
*   **Usage**: Ideal for day-to-day development and CI/CD pipelines.

### 2. Full File System Scan
Activated via `fullLicenseScan.set(true)`. In this mode, Kreate runs `trivy fs --scanners license --license-full` on the project directory.
*   **Pros**: Detects licenses not only in dependency lists but also in source files, Markdown documents, text files, and dedicated `LICENSE` files within your project.
*   **Usage**: Recommended for final release audits or projects with strict compliance requirements.

## Configuration Example

```kotlin
kreate {
    project {
        trivy {
            enabled.set(true)
            license {
                // Fail the build if a forbidden license is found
                failOnForbidden.set(true)

                // Define which severities should be reported
                severity.set(listOf(
                    LicenseSeverity.CRITICAL,
                    LicenseSeverity.HIGH,
                    LicenseSeverity.UNKNOWN
                ))

                // Licenses that are explicitly allowed, even if Trivy flags them
                ignoredLicenses.set(listOf("MIT", "Apache-2.0"))

                // Enable deep scan of the whole project directory
                fullLicenseScan.set(false)
            }
        }
    }
}
```

## License Evaluation Details

Trivy categorizes licenses based on its internal rules into several severity levels.

| Severity | Meaning |
| :--- | :--- |
| `CRITICAL` | Highly restrictive licenses (e.g., AGPL) that often pose a high risk. |
| `HIGH` | Restrictive licenses (e.g., GPL, LGPL). |
| `MEDIUM` | Licenses with certain conditions (e.g., MPL). |
| `LOW` | Permissive licenses (e.g., BSD). |
| `UNKNOWN` | Licenses that Trivy cannot uniquely identify or categorize. |

<warning>
Trivy's license classification is a technical aid and does not replace legal advice. A <code>CRITICAL</code> finding merely indicates that the license is potentially risky according to the Trivy database.
</warning>

## Tips & Best Practices

*   **Lockfile Management**: Ensure you are using Gradle lockfiles (`dependencyLocking { lockAllConfigurations() }`) for the most accurate and fastest results in default mode.
*   **Handling "Unknown"**: Licenses are often marked as `UNKNOWN` if metadata in Maven Central is incomplete. Verify these manually and add them to `ignoredLicenses` if they are acceptable.
*   **Aggressive Compliance**: For highly regulated projects, use `fullLicenseScan.set(true)` to ensure that no stray license files in subdirectories or vendor folders are missed.
*   **Gradual Adoption**: For existing projects, start with `failOnForbidden.set(false)` to identify issues without breaking the build, then fix or ignore them before enforcing compliance.

<tip>
Use <code>./gradlew trivyLicenseScan</code> to run only the license check without starting the full build cycle.
</tip>
